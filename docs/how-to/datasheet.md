# How to generate the feature datasheet

## What is the datasheet?

The feature datasheet is an IC-datasheet-style document for handover to the integration team. It describes what the feature actually does — characterised performance, interface parameters, absolute maximum ratings, application notes, and errata.

It is generated from source files in `features/<feature>/datasheet/` and is never manually edited.

## Source files (you edit these)

| File | What to fill in |
|---|---|
| `datasheet/specs.yaml` | Characterised min/nom/max values for all interfaces and performance parameters |
| `datasheet/application-notes.md` | Typical application, configuration guidance, layout recommendations |
| `datasheet/errata.md` | Known issues against specific hardware revisions |

## Generated files (do not edit)

| File | How it is created |
|---|---|
| `<feature>-datasheet.md` | Generated by `/datasheet` command or auto-generated at CDR merge |
| `<feature>-datasheet.pdf` | Generated at the same time as the Markdown |

## When to use `/datasheet`

Run `/datasheet` in a PR comment any time you want to regenerate the output after editing the source files. The updated Markdown and PDF are committed to `main` automatically.

## Lifecycle

1. **CDR merge** — CI generates the datasheet stub automatically if it does not already exist. Never overwrites an existing datasheet.
2. **Between CDR and TRR** — fill in `specs.yaml` and `application-notes.md`. Run `/datasheet` to preview.
3. **Before TRR** — the TRR checklist requires all `[COMPLETE BEFORE TRR]` placeholders to be replaced and the datasheet committed.
4. **TRR merge** — CI does not touch the datasheet. The version committed before TRR is the released version.

## Requirements vs specs

`requirements/interface-requirements.yaml` contains what the design **must** do.
`datasheet/specs.yaml` contains what the design **actually does**.

Do not copy values between these files. They describe the same interfaces from different perspectives.
