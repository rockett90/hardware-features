# features

> One directory per hardware feature — KiCad projects, schematics, PCB layouts, analysis, and documentation all live here.

---

## What is this directory?

Each subdirectory in `features/` represents a single hardware feature. Features are created using the `init/<feature>` PR process and scaffolded automatically by CI — do not create feature directories manually.

---

## Creating a new feature

Do not create a feature directory by hand. Instead, follow the init PR process:

1. Create a branch `init/<feature>`.
2. Push the branch so `init-branch-setup.yml` scaffolds the full directory structure automatically.
3. Pull the scaffold commit, add the required seed files, then open and merge the PR.

`init-feature.yml` is currently disabled. The scaffold is committed to the `init/<feature>` branch when it is first pushed, before the PR is opened.

See [docs/how-to/init-feature.md](../docs/how-to/init-feature.md) for step-by-step instructions.

---

## Standard feature directory structure

After `init-branch-setup.yml` runs, every feature directory contains:

```
features/<feature>/
├── kicad/
│   ├── <feature>.kicad_pro   — KiCad project file
│   ├── <feature>.kicad_sch   — Top-level schematic
│   └── <feature>.kicad_pcb   — PCB layout
├── .kibot.yml                — KiBot render configuration (references config/kibot/base.kibot.yml)
├── simulations/
│   └── models/               — SPICE models
├── calculations/             — Hand calculations and spreadsheets
├── analysis/
│   ├── mtbf/                 — MTBF analysis
│   ├── stress/               — Component stress analysis
│   ├── thermal/              — Thermal analysis
│   └── doe/                  — Design of experiments
├── bom/                      — Bill of materials
├── bring-up/
│   └── scripts/              — Bring-up scripts (import from bench/)
├── circuit-mods/             — Approved circuit modifications
├── production/
│   ├── fptcs/                — Feature Production Test and Calibration Specification (fptcs.yaml + fptcs-notes.md)
│   ├── test-programs/        — Automated test programs
│   └── aoi/                  — AOI inspection files
├── decisions/                — Design decision records (DDRs)
├── requirements/             — Feature, interface, and verification requirements
├── reviews/                  — Gate evidence (auto-created by CI), library.lock, and TRR-scaffolded `external-references.md` for SharePoint/IV&V links — see reviews/README.md
└── ci-results/               — Artefacts generated by CI (auto-populated, do not edit manually)
```

---

## The one-feature rule

> ⚠️ Warning: A single PR must only touch KiCad files (`.kicad_sch`, `.kicad_pcb`, `.kicad_pro`) in **one** feature directory. CI (`pr-checks.yml`) will reject any PR that modifies files in multiple feature directories.

> ⚠️ Warning: Never manually resolve conflicts in `.kicad_sch` or `.kicad_pcb` files. KiCad file format is not merge-safe — conflicts will produce silently corrupted files.

---

## KiCad project files

The KiCad project files for a feature live in `kicad/`:

```
features/<feature>/kicad/<feature>.kicad_pro
features/<feature>/kicad/<feature>.kicad_sch   ← top-level schematic
features/<feature>/kicad/<feature>.kicad_pcb
```
